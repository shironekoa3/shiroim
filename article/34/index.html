SpringSecurity笔记

发表于2023-03-19|更新于2023-03-19|笔记
|字数总计:1.3k|阅读时长:5分钟

前言

在 Web 开发中，网络安全越来越被重视。在这方面比较出名的就是 SpringSecurity 和 Shiro 这两个框架了。

Shiro 框架相对更轻量级，使用更方便，在一些小项目中比较常用。

SpringSecurity 的配置繁杂，在 SpringBoot 之前的用户并不多。

SpringBoot 诞生之后，SpringSecurity 的配置问题有了很好的处理方案，现在的用户越来越多。

引入

由于 SpringBoot 极大地简化了各个框架的配置，现在引入 SpringSecurity 变得十分简单。

只需要引入 spring-boot-starter-security 依赖即可。

引入依赖之后，启动项目时会生成随机密码，此时访问任何接口都会强制跳转到内置的登陆页，使用用户名 user 和控制台的随机密码登陆之后才能正常访问。

认证

SpringSecurity 的原理就是一个过滤器链。

默认认证用户名是 user，默认密码每次启动会在控制台随机生成。

我们需要将默认的认证方式改成我们自己的。要实现这个功能，就需要熟悉认证相关的过滤器：UsernamePasswordAuthenticationFilter

认证流程：

  1. UsernamePasswordAuthenticationFilter 首先会拿到传入的用户名和密码，并封装成一个 Authentication 对象。
  2. 调用 UserDetailsService 接口中的 loadUserByUsername 方法。此方法会查询目标用户的用户名、密码以及其对应的权限，封装成 UserDetails 对象并返回。
  3. 通过 PasswordEncoder 中的 matches 方法比对 Authentication 中的用户名密码和 UserDetails 中的是否一致。如果密码正确，则把 UserDetails 中的权限设置到 Authentication 中。
  4. 如果认证成功，会通过 SecurityContextHolder.getContext().setAuthentication() 将 Authentication 对象保存。其他过滤器通过 SecurityContextHolder 获取当前用户信息。

从上述的流程来看，要替换成我们自己的认证，只需要重写 loadUserByUsername 方法，使其从数据库中获取用户。然后在容器中放入一个 PasswordEncoder 接口的实现类，用来比对密码是否一致即可。
